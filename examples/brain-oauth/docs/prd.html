🧠 Brain OAuth 2.0 中间层

最终产品需求与技术设计 · 标准 OAuth 2.0 授权服务器 · 开发者控制台 · 无服务器就绪

版本 5.0 (最终版) | 最后更新 2026-05-22 | 状态 可直接开发

本文档是 Brain OAuth 2.0 授权服务器的完整蓝图。它将 Brain 现有非标准令牌接口包装成符合 RFC 6749 的授权服务器，并提供开发者控制台，支持第三方应用安全接入 Brain 用户体系。

1. 项目概述

核心目标：构建一个轻量级 OAuth 2.0 中间层，对外暴露标准授权端点、令牌端点和用户信息端点，对内调用 Brain 原生 API（密码换 token，token 换 access_token）。中间层自身不签发 token，完全复用 Brain 下发的 access_token 和 refresh_token，并允许开发者自助注册应用（获得 client_id/secret）。

技术栈：Next.js 14 (App Router) + TypeScript，Supabase (PostgreSQL) 作为数据库，iron-session 管理用户会话，bcryptjs 哈希 client_secret，jose 用于 JWT（可选，仅用于内部），部署于 Vercel 等无服务器环境。

2. 路由与渲染策略

所有路由严格区分 页面组件（.tsx）和 API 路由（route.ts）。

2.1 页面路由（用户界面）

  • / → 首页（品牌介绍 + 动态按钮），SSR 或静态生成（读取 session 判断登录状态）
  • /login → 登录页，SSR（无数据依赖，可静态但无妨）
  • /oauth/authorize → 授权确认页，SSR（需要校验 client 信息）
  • /developer/apps → 我的应用列表，SSR（依赖登录用户，查询数据库）
  • /developer/apps/new → 创建应用表单，SSR（仅表单，可静态）
  • /developer/apps/[clientId]/edit → 编辑应用，SSR（需加载当前应用数据）

2.2 API 路由

  • 标准 OAuth 端点（对外）：
    • POST /oauth/token → 令牌交换
    • GET /userinfo → 用户信息
  • 内部 API（供前端页面调用）：
    • POST /api/brain/verify → 验证 Brain 凭证，建立会话
    • POST /api/oauth/consent → 用户同意后生成授权码
    • GET /api/clients → 获取当前用户的应用列表
    • POST /api/clients → 创建新应用
    • GET /api/clients/:clientId → 获取应用详情
    • PUT /api/clients/:clientId → 更新应用信息
    • DELETE /api/clients/:clientId → 删除应用
    • POST /api/clients/:clientId/rotate-secret → 重置密钥

2.3 登录跳转逻辑

  • /login 页面从 URL query 中读取 redirect 参数。
  • 登录成功后：
    • 如果 redirect 存在且为本站内部路径（以 / 开头），则跳转至该路径。
    • 否则默认跳转至 /developer/apps。
  • 未登录用户访问受保护页面（/oauth/authorize、/developer/*）时，中间件自动重定向到 /login?redirect=原始路径。

3. 开发者控制台（核心功能）

第三方开发者使用 Brain 账号登录后，可自助创建 OAuth 应用，获得 client_id 和 client_secret。

3.1 应用列表页 (/developer/apps)

  • 展示当前用户所有应用的 client_id、名称、回调地址（简略）、创建时间。
  • 操作按钮：编辑、重置密钥、删除。

3.2 创建应用页 (/developer/apps/new)

  • 表单字段：应用名称（必填）、重定向 URI（必填，支持多行输入）、主页 URL（可选）、Logo 地址（可选）。
  • 提交后生成 client_id（UUID v4）和 client_secret（32 字节随机，bcrypt 哈希存储）。
  • 显示成功提示并展示 client_id 和 client_secret（明文仅显示一次，提供复制按钮）。

3.3 编辑应用页 (/developer/apps/[clientId]/edit)

  • 可修改应用名称、回调地址、主页 URL 等。
  • 提供“重置密钥”按钮：生成新 client_secret，展示新明文一次，旧 secret 立即失效。
  • 提供“删除应用”按钮：二次确认后删除，同时清理关联的授权码和 refresh_token 映射。

4. 数据库设计（Supabase）

所有表均需启用行级安全（RLS），服务角色拥有全部权限，普通用户仅能读取/操作自己拥有的记录（client 表通过 owner_user_id 限制）。

4.1 oauth_clients

CREATE TABLE oauth_clients (
    id                SERIAL PRIMARY KEY,
    client_id         TEXT UNIQUE NOT NULL,
    client_secret_hash TEXT NOT NULL,
    client_name       TEXT NOT NULL,
    client_uri        TEXT,
    logo_uri          TEXT,
    redirect_uris     TEXT[] NOT NULL,
    grant_types       TEXT[] DEFAULT '{authorization_code,refresh_token}',
    scopes            TEXT[] DEFAULT '{openid,profile,email}',
    confidential      BOOLEAN DEFAULT true,
    owner_user_id     INTEGER NOT NULL,
    created_at        TIMESTAMPTZ DEFAULT now(),
    updated_at        TIMESTAMPTZ DEFAULT now()
);

4.2 oauth_authorization_codes

CREATE TABLE oauth_authorization_codes (
    code          TEXT PRIMARY KEY,
    client_id     TEXT NOT NULL REFERENCES oauth_clients(client_id) ON DELETE CASCADE,
    user_id       INTEGER NOT NULL,
    redirect_uri  TEXT NOT NULL,
    scope         TEXT,
    expires_at    TIMESTAMPTZ NOT NULL,
    used          BOOLEAN DEFAULT false,
    created_at    TIMESTAMPTZ DEFAULT now()
);

4.3 oauth_refresh_tokens（Brain 原生 refresh_token 映射）

CREATE TABLE oauth_refresh_tokens (
    id            SERIAL PRIMARY KEY,
    refresh_token TEXT NOT NULL UNIQUE,
    client_id     TEXT NOT NULL REFERENCES oauth_clients(client_id),
    user_id       INTEGER NOT NULL,
    expires_at    TIMESTAMPTZ NOT NULL,
    revoked       BOOLEAN DEFAULT false,
    created_at    TIMESTAMPTZ DEFAULT now()
);

4.4 oauth_user_credentials（用户长期凭证）

CREATE TABLE oauth_user_credentials (
    user_id             INTEGER PRIMARY KEY,
    brain_refresh_token TEXT,   -- 加密存储
    brain_session_token TEXT,   -- 备选
    updated_at          TIMESTAMPTZ DEFAULT now()
);
🔐 refresh_token 和 brain_refresh_token 必须使用 AES-256-GCM 加密，密钥来自环境变量 ENCRYPTION_KEY。

5. Brain 用户系统集成

5.1 登录流程

  1. 前端提交 email/password 到 /api/brain/verify。
  2. 调用 Brain /auth/token.json 获取 session token。
  3. 用 session token 调用 Brain /auth/access_token 获取 access_token + refresh_token（24h）。
  4. 调用 Brain /users/me.json 获取用户信息（user_id, email, name）。
  5. 将 refresh_token 加密存入 oauth_user_credentials（user_id 为主键）。
  6. 建立中间层 session（存储 user_id, email, name）。

5.2 令牌端点中换取 access_token

当第三方应用使用授权码换 token 时，中间层从 oauth_user_credentials 获取该用户的加密 refresh_token，解密后调用 Brain /auth/access_token（需确认 Brain 是否支持用 refresh_token 换取新 token；若不支持，则使用存储的 session_token 重新调用）。

⚠️ 需要根据 Brain 实际接口行为调整：若 /auth/access_token 仅接受 session token，则应在用户登录时存储 session_token，并在每次令牌请求时使用它获取新 access_token（session_token 需长期有效）。

6. 标准 OAuth 2.0 端点实现

6.1 GET /oauth/authorize（页面）

  • 服务端组件读取 query 参数，校验 client_id、redirect_uri 白名单。
  • 检查用户登录状态（session），未登录则重定向到 /login?redirect=当前URL。
  • 已登录则渲染确认页面，显示应用名称和请求的 scope。
  • 用户提交同意后，调用 POST /api/oauth/consent 生成授权码并重定向到客户端。

6.2 POST /oauth/token（API）

  • 支持 grant_type=authorization_code 和 refresh_token。
  • 验证 client 凭证（Basic Auth 或 body 参数）。
  • 对于 authorization_code：验证授权码（存在、未过期、未使用、redirect_uri 匹配），标记已使用，然后用用户的长期凭证从 Brain 获取新 token，返回给客户端。
  • 对于 refresh_token：验证数据库中的 refresh_token 有效，调用 Brain 刷新，返回新 token 并更新存储。

6.3 GET /userinfo（API）

  • 提取 Authorization: Bearer <access_token>，转发到 Brain /users/me.json。
  • 若 Brain 返回 401，则响应 { "error": "invalid_token" }，HTTP 401。
  • 成功则返回用户信息（sub, email, name, roles 等）。

7. 错误码规范（符合 OAuth 2.0）

7.1 授权端点错误（重定向）

  • invalid_request：缺少 response_type 或参数不合法。
  • unauthorized_client：redirect_uri 不匹配客户端注册值。
  • access_denied：用户拒绝授权。
  • unsupported_response_type：仅支持 code。
  • invalid_scope：scope 超出客户端允许范围。
  • server_error：中间层内部错误。

7.2 令牌端点错误（JSON）

  • invalid_client (401)：client 认证失败。
  • invalid_grant (400)：授权码无效/过期/已使用，或 refresh_token 无效。
  • unsupported_grant_type (400)：grant_type 不支持。
  • invalid_request (400)：请求格式错误。

7.3 内部错误码（仅日志）

  • BRAIN_API_FAILED、REFRESH_TOKEN_NOT_FOUND、USER_SESSION_EXPIRED 等。

8. 安全与无服务器适配

  • 所有 Cookie：HttpOnly; Secure; SameSite=Lax。
  • 授权码一次性，5 分钟过期。
  • client_secret 使用 bcrypt（强度 12）哈希存储。
  • 数据库敏感字段 AES-256-GCM 加密。
  • Supabase 连接池：postgres://...?pgbouncer=true。
  • 使用 @supabase/supabase-js 或 drizzle-orm 以降低冷启动。

9. 开发顺序建议

  1. 初始化 Next.js + TypeScript + Tailwind 项目，配置 Supabase。
  2. 实现 Supabase 表及 RLS 策略。
  3. 实现登录功能：/api/brain/verify + /login 页面，测试 Brain 认证。
  4. 实现开发者控制台：/api/clients/* + /developer/apps 相关页面。
  5. 实现授权流程：/oauth/authorize 页面 + /api/oauth/consent + 授权码存储。
  6. 实现令牌端点 /oauth/token 和用户信息端点 /userinfo。
  7. 集成测试：模拟第三方客户端完成完整授权码流程。
  8. 部署到 Vercel，配置环境变量。
