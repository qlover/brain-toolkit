2. 页面路由详解

2.1 登录页面 /login

  • 职责：提供邮箱/密码表单，通过 AJAX 调用 /api/brain/verify 完成 Brain 认证。
  • 交互：成功后从 URL query 中获取 redirect 参数（通常是 /oauth/authorize?...），并跳转回该地址；若无则跳转 /dashboard/apps。
  • 样式：简洁卡片风格，包含加载状态和错误提示。

2.2 授权确认页面 /oauth/authorize

  • 职责：展示第三方应用名称、请求的权限范围，让用户确认授权。
  • 前置条件：用户必须已登录（中间层 session）。未登录时自动重定向到 /login?redirect=当前路径。
  • 实现逻辑：
    1. 服务端组件中读取 URL query 参数 client_id、redirect_uri、response_type、state、scope。
    2. 验证 response_type=code，client_id 在数据库中存在且 redirect_uri 匹配白名单。
    3. 渲染确认表单，显示应用名称（从数据库获取）和请求的 scope 描述。
    4. 用户点击“同意授权”后，提交表单到 /api/oauth/consent（携带原始 query 参数）。

2.3 开发者控制台页面

  • 访问控制：所有 /dashboard/* 路由需要中间件校验用户登录，未登录重定向到 /login。
  • 应用列表页 (/dashboard/apps)：调用 GET /api/clients 展示当前用户的所有 OAuth 应用，提供“创建应用”、“编辑”、“重置密钥”、“删除”按钮。
  • 创建应用页 (/dashboard/apps/new)：表单填写应用名称、重定向 URI（可多个）、主页 URL、Logo 地址。提交后调用 POST /api/clients，成功后展示新生成的 client_id 和 client_secret（仅显示一次，强制复制）。
  • 编辑应用页 (/dashboard/apps/[clientId]/edit)：加载现有信息，可修改名称、回调地址等。提供“重置密钥”和“删除应用”按钮，操作前需二次确认。

3. API 路由详解

3.1 标准 OAuth 2.0 端点（对外）

POST /oauth/token

  • 请求格式: application/x-www-form-urlencoded 或 application/json。支持 Basic Auth（client_id:client_secret）或参数内传递。
  • 支持的 grant_type: authorization_code、refresh_token。
  • 处理逻辑:
    1. 验证客户端凭证，比对数据库中的 client_secret_hash（bcrypt）。
    2. 若为 authorization_code：验证授权码是否存在、未过期、未使用且 redirect_uri 匹配；标记授权码为已使用；从 oauth_user_credentials 获取该用户的 Brain 长期凭证（refresh_token 或 session_token）；调用 Brain /auth/access_token 换取新的 access_token 和 refresh_token；更新数据库中存储的 refresh_token；返回响应。
    3. 若为 refresh_token：从数据库查找该 refresh_token 是否有效且关联正确的 client；调用 Brain 刷新接口（若支持），否则使用用户存储的 session token 重新获取；返回新 token。
  • 响应示例: 
    {
  "access_token": "brain_access_token",
  "token_type": "Bearer",
  "expires_in": 86400,
  "refresh_token": "brain_refresh_token"
}

GET /userinfo

  • 请求头: Authorization: Bearer <access_token>（Brain 原生 access_token）。
  • 实现: 直接转发到 Brain /users/me.json 并携带该 token。若 Brain 返回 401，则返回 OAuth 标准错误 invalid_token。成功则返回用户信息（sub、email、name、roles 等）。

3.2 内部 API（仅供中间层前端调用）

POST /api/brain/verify

  • 请求体: { email, password }。
  • 处理:
    1. 调用 Brain /auth/token.json 获取 session_token。
    2. 使用 session_token 调用 Brain /auth/access_token 换取 access_token 和 refresh_token。
    3. 同时调用 /users/me.json 获取用户基本信息（id, email, name）。
    4. 将 refresh_token 加密存储到 oauth_user_credentials 表（user_id 为主键）。
    5. 建立中间层 session（存储 user_id, email, name）。
    6. 返回成功状态。

POST /api/oauth/consent

  • 请求体: 包含授权确认页面的所有 query 参数（client_id, redirect_uri, response_type, state, scope 等）。
  • 处理:
    1. 验证当前用户已登录（从 session 获取 user_id）。
    2. 再次验证 client_id 和 redirect_uri 的合法性。
    3. 生成授权码（crypto.randomBytes(32).toString('hex')）。
    4. 存储到 oauth_authorization_codes，关联 user_id、client_id、redirect_uri、scope，过期时间 5 分钟。
    5. 返回重定向 URL: ${redirect_uri}?code=${code}&state=${state}。

客户端管理 API（/api/clients）

  • GET /api/clients: 返回当前用户的所有客户端（基于 session 中 user_id 匹配 owner_user_id）。
  • POST /api/clients: 创建新客户端，字段：client_name、redirect_uris（数组）、client_uri、logo_uri。自动生成 client_id（UUID v4）和 client_secret（32 字节随机，bcrypt 哈希存储）。返回 client_id 和明文 client_secret（仅此一次）。
  • GET /api/clients/:clientId: 获取详情，不返回 secret。
  • PUT /api/clients/:clientId: 更新名称、回调地址等，client_id 不可变。
  • DELETE /api/clients/:clientId: 删除客户端，级联失效相关授权码和 refresh_token 映射。
  • POST /api/clients/:clientId/rotate-secret: 生成新 secret，更新哈希，返回明文新 secret。
  • 所有接口均需验证登录用户是客户端所有者（owner_user_id === session.user_id）或管理员。

4. 数据库表设计（Supabase）

4.1 oauth_clients

CREATE TABLE oauth_clients (
    id                SERIAL PRIMARY KEY,
    client_id         TEXT UNIQUE NOT NULL,
    client_secret_hash TEXT NOT NULL,
    client_name       TEXT NOT NULL,
    client_uri        TEXT,
    logo_uri          TEXT,
    redirect_uris     TEXT[] NOT NULL,
    grant_types       TEXT[] DEFAULT '{authorization_code,refresh_token}',
    scopes            TEXT[] DEFAULT '{openid,profile,email}',
    confidential      BOOLEAN DEFAULT true,
    owner_user_id     INTEGER NOT NULL,
    created_at        TIMESTAMPTZ DEFAULT now(),
    updated_at        TIMESTAMPTZ DEFAULT now()
);

4.2 oauth_authorization_codes

CREATE TABLE oauth_authorization_codes (
    code          TEXT PRIMARY KEY,
    client_id     TEXT NOT NULL REFERENCES oauth_clients(client_id) ON DELETE CASCADE,
    user_id       INTEGER NOT NULL,
    redirect_uri  TEXT NOT NULL,
    scope         TEXT,
    expires_at    TIMESTAMPTZ NOT NULL,
    used          BOOLEAN DEFAULT false,
    created_at    TIMESTAMPTZ DEFAULT now()
);

4.3 oauth_refresh_tokens（存储 Brain 原生 refresh_token 的映射）

CREATE TABLE oauth_refresh_tokens (
    id            SERIAL PRIMARY KEY,
    refresh_token TEXT NOT NULL UNIQUE,   -- Brain refresh_token (加密)
    client_id     TEXT NOT NULL REFERENCES oauth_clients(client_id),
    user_id       INTEGER NOT NULL,
    expires_at    TIMESTAMPTZ NOT NULL,
    revoked       BOOLEAN DEFAULT false,
    created_at    TIMESTAMPTZ DEFAULT now()
);

4.4 oauth_user_credentials（存储用户长期凭证）

CREATE TABLE oauth_user_credentials (
    user_id             INTEGER PRIMARY KEY,
    brain_refresh_token TEXT,   -- 加密存储 Brain 原生 refresh_token
    brain_session_token TEXT,   -- 备选方案
    updated_at          TIMESTAMPTZ DEFAULT now()
);
🔐 注意：refresh_token 和 brain_refresh_token 字段必须使用应用层 AES-256-GCM 加密，密钥从环境变量获取。

5. 错误码规范（按端点分组）

5.1 授权页面错误（重定向错误）

当参数校验失败时，重定向到 redirect_uri?error=xxx&error_description=...

  • invalid_request：缺少 response_type 或参数格式错误。
  • unauthorized_client：redirect_uri 不匹配。
  • access_denied：用户拒绝授权。
  • unsupported_response_type：response_type 不是 code。
  • invalid_scope：scope 不在客户端允许范围。

5.2 令牌端点错误（JSON）

  • invalid_client (401)：client_id 无效或 secret 错误。
  • invalid_grant (400)：授权码无效/过期/已使用，或 refresh_token 无效。
  • unsupported_grant_type (400)：grant_type 不支持。
  • invalid_request (400)：缺少必要参数。

5.3 用户信息端点错误

  • invalid_token (401)：access_token 无效或过期。

6. 安全与无服务器适配要点

  • 所有 Cookie 使用 HttpOnly; Secure; SameSite=Lax。
  • 授权码一次性使用，5 分钟过期。
  • client_secret 使用 bcrypt（强度 12）存储，永不记录明文。
  • 数据库敏感字段使用 AES-256-GCM 加密（ENCRYPTION_KEY）。
  • Supabase 连接池配置：postgres://...?pgbouncer=true 减少冷启动延迟。
  • 使用 @supabase/supabase-js 或 drizzle-orm 避免 Prisma 冷启动开销。
  • 在 Vercel 部署时，将 BRAIN_API_BASE 等环境变量配置完整。
