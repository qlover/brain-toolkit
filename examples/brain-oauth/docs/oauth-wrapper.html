📦 Next.js OAuth Wrapper

通用 OAuth 2.0 授权服务器套件 · 基于 Next.js + 任意用户系统适配器

1. 项目定位

Next.js OAuth Wrapper 是一个可插拔的 OAuth 2.0 / OIDC 授权服务器中间件。它封装了标准的授权端点、令牌端点、用户信息端点、PKCE 支持、客户端管理后台及数据库持久化。你只需实现一个轻量的用户认证适配器（User Adapter），即可让任何现有的用户系统（如旧版 API、自建用户表、LDAP、甚至 Brain User API）拥有标准 OAuth 能力。

✨ 专为无服务器环境设计（Vercel / Netlify），使用 Supabase（PostgreSQL）存储客户端与授权码，支持公共客户端（PKCE）和机密客户端。

2. 核心特性

  • ✅ 标准 OAuth 2.0 授权码流程 + PKCE（RFC 7636）
  • ✅ 公共客户端（无需 client_secret）与机密客户端（需 client_secret）双模式
  • ✅ OpenID Connect 用户信息端点（/userinfo）
  • ✅ Refresh Token 自动刷新
  • ✅ 开发者控制台（应用注册、管理 client_id/secret）
  • ✅ 中间件保护 /oauth/authorize 和 /developer
  • ✅ 支持任意用户系统：通过 Adapter 接口，适配登录验证、用户信息获取
  • ✅ 数据库使用 Supabase（可替换为其他 PostgreSQL）
  • ✅ 国际化（i18n）与主题切换（亮色/暗色）就绪

4. 用户适配器接口（关键）

为了让包装器适配任意用户系统，你需要实现以下两个核心函数：

// lib/adapter/user-adapter.ts
export interface UserAdapter {
  /**
   * 验证用户凭证，返回用户唯一标识（如 user_id）
   * @param email 邮箱或用户名
   * @param password 明文密码
   * @returns 用户对象（至少包含 id, email, name）或 null
   */
  verifyCredentials(email: string, password: string): Promise<{ id: string | number; email: string; name: string } | null>;

  /**
   * 根据用户 ID 获取用户资料（用于 /userinfo 或颁发 token 时的 claim）
   * @param userId 用户唯一标识
   * @returns 用户资料对象
   */
  getUserInfo(userId: string | number): Promise<{ sub: string; email: string; name: string; [key: string]: any }>;
}
📌 示例：适配 Brain User API 
export const brainAdapter: UserAdapter = {
  async verifyCredentials(email, password) {
    const res = await fetch('https://api.brain.com/login', { method: 'POST', body: JSON.stringify({ email, password }) });
    if (!res.ok) return null;
    const data = await res.json();
    return { id: data.user_id, email: data.email, name: data.name };
  },
  async getUserInfo(userId) {
    const res = await fetch(`https://api.brain.com/users/${userId}`);
    const user = await res.json();
    return { sub: String(user.id), email: user.email, name: user.name };
  }
};

5. 数据库表结构（Supabase）

包装器使用以下表，所有表均通过 Supabase 迁移自动创建：

  • oauth_clients — 存储客户端（client_id, client_secret_hash, redirect_uris, owner_user_id, confidential, 等）
  • oauth_authorization_codes — 授权码（code, client_id, user_id, redirect_uri, code_challenge, code_challenge_method, expires_at, used）
  • oauth_refresh_tokens — 刷新令牌映射（refresh_token, client_id, user_id, expires_at）
  • oauth_user_credentials — 可选，用于存储用户长期凭证（仅当用户系统不支持 refresh_token 时需要）

具体 SQL 见项目 migrations/ 目录。

7. 标准 OAuth 端点说明

7.1 授权端点 GET /oauth/authorize

参数：client_id, redirect_uri, response_type=code, state, scope, code_challenge, code_challenge_method
行为：校验客户端，展示授权确认页（需登录），用户同意后生成授权码并重定向。

7.2 令牌端点 POST /oauth/token

支持 grant_type=authorization_code 和 refresh_token。
对于公共客户端：验证 PKCE code_verifier；对于机密客户端：验证 client_secret。
响应：{ access_token, token_type, expires_in, refresh_token }。

7.3 用户信息端点 GET /userinfo

需要 Authorization: Bearer <access_token>，返回适配器提供的用户资料。

8. 开发者控制台 API

全部位于 /api/clients 下，需要登录用户（通过 session 识别 owner_user_id）。

  • GET /api/clients — 列出当前用户的应用
  • POST /api/clients — 创建应用（自动生成 client_id 和 client_secret）
  • GET /api/clients/:clientId — 获取详情
  • PUT /api/clients/:clientId — 更新回调地址等
  • DELETE /api/clients/:clientId — 删除应用
  • POST /api/clients/:clientId/rotate-secret — 重置密钥

9. 安全与无服务器优化

  • 所有 cookie 使用 HttpOnly; Secure; SameSite=Lax。
  • 授权码一次性，5 分钟过期。
  • client_secret 使用 bcrypt（强度 12）哈希存储。
  • Supabase 连接池启用 pgbouncer=true。
  • 使用 @supabase/supabase-js 直连，避免 Prisma 冷启动开销。
  • 中间件仅保护必要路由，公开路径：/login、/oauth/token、/userinfo、/api/brain/verify 等。

10. 如何使用这个包装器

步骤 1：克隆模板并安装依赖

git clone https://github.com/your-org/nextjs-oauth-wrapper
cd nextjs-oauth-wrapper
npm install

步骤 2：配置环境变量（.env.local）

SUPABASE_URL=...
SESSION_SECRET=...
# 其他变量

步骤 3：实现用户适配器

在 lib/adapter/user-adapter.ts 中实现 verifyCredentials 和 getUserInfo，对接你自己的登录接口。

步骤 4：运行数据库迁移

npm run migrate:up

步骤 5：启动开发服务

npm run dev

步骤 6：访问开发者控制台

http://localhost:3000/developer/apps，注册第一个 OAuth 客户端。

步骤 7：测试 OAuth 流程

使用上一步获得的 client_id 发起授权请求，验证完整流程。

11. 扩展性

  • 支持自定义 scope：可在适配器中根据 scope 过滤返回的用户信息字段。
  • 支持 OpenID Connect Discovery：添加 /.well-known/openid-configuration 端点。
  • 多租户：通过修改 oauth_clients 表增加 tenant_id 字段实现。
  • 企业级审计：可接入日志服务记录每次 token 颁发和用户授权。
